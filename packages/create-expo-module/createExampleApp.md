# `createExampleApp.ts` вЂ” Step-by-Step Explanation

This file is responsible for creating and configuring an `example/` app inside a newly scaffolded Expo module. The example app is a minimal Expo project that imports the new module, giving developers a ready-to-run environment for testing native code.

---

## Entry point: `createExampleApp(data, targetDir, packageManager)`

Called from `create-expo-module.ts` after the module template files have been rendered. Only runs for **remote** (non-local) modules when `--no-example` is not passed.

**Parameters:**
- `data` вЂ” substitution data from the scaffolding step (slug, package name, etc.)
- `targetDir` вЂ” the root directory of the newly created module (e.g. `/path/to/my-module`)
- `packageManager` вЂ” detected package manager (`npm`, `yarn`, `pnpm`, or `bun`)

---

## Step 0: Guard вЂ” does the template include an `example/` folder?

```
targetDir/
в””в”Ђв”Ђ example/     в†ђ rendered from expo-module-template
```

```ts
if (!fs.existsSync(appTargetPath)) {
  return;
}
```

The `example/` directory is part of the `expo-module-template` tarball. If it's absent (e.g. a future template without it), the function exits immediately and skips all further steps.

---

## Step 1: Initialize the example app

**Runs:** `<packageManager> create expo-app <slug>-example --template expo-template-blank-typescript@latest --yes`

The command creates a brand-new Expo app in a **temporary directory** next to `example/`:

```
targetDir/
в”њв”Ђв”Ђ example/                в†ђ template files (from expo-module-template)
в””в”Ђв”Ђ my-module-example/      в†ђ freshly created by create expo-app  в†ђ TEMP
```

The slug for the new app is `<module-slug>-example` (e.g. `my-module-example`).

**Template version:** `expo-template-blank-typescript@latest`, or `@next` when `EXPO_BETA=true`.

**`createCommand` helper:** npm requires an extra `--` separator before positional arguments when run via `npm create`, so the command is built differently per package manager:
- `yarn/pnpm/bun`: `create expo-app my-module-example --template ... --yes`
- `npm`: `create expo-app -- my-module-example --template ... --yes`

---

## Step 2: Configure the example app

This step merges the template's `example/` files into the freshly created app, then cleans up.

### 2a. Move template files into the temp app

```ts
await moveFiles(appTargetPath, appTmpPath);
// moves: example/* в†’ my-module-example/*
```

`moveFiles` copies every file from `example/` on top of `my-module-example/`, **overwriting** any conflicts. This is how the module template customises the example app вЂ” it ships its own `App.tsx`, config files, etc. that replace the defaults from `create expo-app`.

The move uses `fs.rename` for performance. If the source and destination are on different filesystems (error code `EXDEV`), it falls back to `fs.cp` + `fs.rm`.

### 2b. Delete the now-empty `example/` directory

```ts
await fs.promises.rm(appTargetPath, { recursive: true, force: true });
```

The original template `example/` directory is removed since its files have been moved into the temp app.

### 2c. Delete the `.git` folder from the temp app

```ts
await fs.promises.rm(path.join(appTmpPath, '.git'), { recursive: true, force: true });
```

`create expo-app` initialises its own git repository. This is removed because the example app should share the module's git repository, not have its own.

### 2d. Rename the temp app to `example/`

```ts
await fs.promises.rename(appTmpPath, appTargetPath);
// my-module-example/ в†’ example/
```

The directory is moved into its final location.

```
targetDir/
в””в”Ђв”Ђ example/     в†ђ final configured app (was my-module-example/)
```

### 2e. Add missing `app.json` fields

```ts
await addMissingAppConfigFields(appTargetPath, data);
```

`expo prebuild` (run in the next step) requires `android.package` and `ios.bundleIdentifier` to be set in `app.json`. The blank TypeScript template doesn't include these, so they're injected from the module's package name:

```json
{
  "expo": {
    "android": { "package": "expo.modules.mymodule.example" },
    "ios":     { "bundleIdentifier": "expo.modules.mymodule.example" }
  }
}
```

The app ID is the module's Android package name with `.example` appended.

## Step 3: Prebuild the example app

```ts
await spawnAsync('npx', ['expo', 'prebuild', '--no-install'], { cwd: exampleAppPath });
```

`expo prebuild` generates the native `android/` and `ios/` project folders from the `app.json` config. `--no-install` skips dependency installation since that happens separately in Step 5.

After this step the directory looks like:

```

в”њв”Ђв”Ђ ios/         в†ђ generated by prebuild
в”њв”Ђв”Ђ App.tsx      в†ђ from module template
в””в”Ђв”Ђ app.json
```

## Step 4: Modify `example/package.json`

```ts
await modifyPackageJson(appTargetPath);
```

Two changes are made to the example app's `package.json`:

### 4a. Set autolinking config

```json
{
  "expo": {
    "autolinking": {
      "nativeModulesDir": ".."
    }
  }
}
```

This tells Expo's autolinking to look for native modules one level up вЂ” i.e. in `targetDir/` вЂ” so it finds and links the module being developed without it needing to be published to npm.

### 4b. Remove unnecessary dependencies

```ts
const DEPENDENCIES_TO_REMOVE = ['expo-status-bar', 'expo-splash-screen'];
```

These packages are added by `expo-template-blank-typescript` but are not needed in the example app. They are deleted from `dependencies` before `npm install` runs.

---

## Step 5: Install dependencies

```ts
await installDependencies(packageManager, appTargetPath);
```

Runs `<packageManager> install` inside `example/`, installing all dependencies including the ones from the template.

### CocoaPods (macOS only)

```ts
if (os.platform() === 'darwin') {
  await podInstall(appTargetPath);
}
```

On macOS, `pod install` is run inside `example/ios/` to link native iOS dependencies. This is skipped on Linux and Windows since iOS development requires macOS.
